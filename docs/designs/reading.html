Building a Rust AT Protocol Client

A comprehensive guide to implementing AT Protocol in Rust, covering XRPC, authentication, and record operations.

Alice Chen

@alice.bsky.social · Mar 15, 2024 · 12 min read

The AT Protocol represents a paradigm shift in how we think about social media infrastructure. Unlike traditional centralized platforms, it separates identity, data storage, and indexing into distinct layers, enabling unprecedented flexibility and user control.

Getting Started with XRPC

At the heart of the AT Protocol is XRPC (XML-RPC evolved), a JSON-based RPC protocol that powers all communication between clients and servers. In Rust, we can implement this using standard HTTP clients with a few ergonomic wrappers. 

use atproto::xrpc::Client;

let client = Client::new("https://bsky.social");
let session = client
    .create_session("handle.bsky.social", "password")
    .await?;

println!("Authenticated as: {}", session.did);

The Client struct handles all the boilerplate of making XRPC requests, including proper lexicon validation and type-safe deserialization. It's built on top of reqwest and leverages Rust's async/await for efficient I/O.

Understanding Lexicons

Lexicons are the schema definitions that govern the AT Protocol. They define the shapes of records, the parameters for queries, and the structure of responses. Having strong types for these is essential for a reliable client implementation.

"Lexicons are to AT Protocol what GraphQL schemas are to GraphQL — they provide the contract that all participants in the network must follow."

In our Rust implementation, we generate types from the official lexicon definitions using a custom code generator. This ensures we're always in sync with the protocol spec and catches breaking changes at compile time.

Authentication with OAuth 2.1

The AT Protocol uses modern OAuth 2.1 with DPoP (Demonstrating Proof-of-Possession) for authentication. This provides several advantages over traditional API keys:

  • No password storage required
  • Fine-grained permission scopes
  • Automatic token rotation
  • Secure binding to client instances via DPoP

Implementing this in Rust requires careful handling of cryptographic operations. We use the jsonwebtoken crate for JWT handling and ring for the DPoP proof generation.

Working with Repositories

User data in AT Protocol is stored in repositories — Merkle tree structures that provide cryptographic verification of data integrity. When you post, like, or follow, you're creating new records in your repository. 

// Creating a new post record
let record = PostRecord {
    text: "Hello, AT Protocol!".to_string(),
    created_at: Utc::now(),
    facets: vec![],
    reply: None,
};

let response = client
    .create_record(&session, "app.bsky.feed.post", record)
    .await?;

The create_record method handles all the complexity of encoding the record in CBOR, computing its CID, and updating the repository's Merkle tree. From the developer's perspective, it feels as simple as a standard API call.

Conclusion

Building a robust AT Protocol client in Rust requires understanding the protocol's unique architecture, but the results are worth it: type-safe, performant, and fully interoperable with the growing ecosystem of ATProto applications.

In future posts, we'll dive deeper into specific areas like implementing custom lexicons, building feed generators, and leveraging the protocol's real-time capabilities through Jetstream.
